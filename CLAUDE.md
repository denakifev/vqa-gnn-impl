# CLAUDE.md

## Назначение проекта

Этот репозиторий предназначен для воспроизводимой открытой реализации статьи **VQA-GNN: Reasoning with Multimodal Knowledge via Graph Neural Networks for Visual Question Answering**.

Текущее состояние репозитория: это общий шаблон проекта на PyTorch и Hydra с демонстрационными компонентами (`ExampleDataset`, `BaselineModel`, `ExampleLoss`, `ExampleMetric`). Реализация VQA-GNN, реальные датасеты VQA и пользовательская документация проекта в репозитории пока отсутствуют. Claude Code должен развивать проект, не выдавая шаблонные примеры за реализацию статьи.

## Контекст проекта

В рамках этого проекта под VQA-GNN понимается базовая модель из статьи, которая объединяет визуальные признаки, представление вопроса и внешние знания в графовую схему рассуждения для предсказания ответа.

Целевая базовая модель для первой стадии:

- это исходный VQA-GNN из статьи, без архитектурных улучшений и без исследовательских ответвлений;
- это воспроизводимая инженерная реализация, а не новая вариация модели;
- это запуск через уже существующий шаблон обучения и инференса в текущем репозитории.

В первую очередь должны быть воспроизведены части статьи, которые образуют основной экспериментальный контур:

- входной путь данных для визуальной, текстовой и knowledge-based частей;
- базовая архитектура VQA-GNN и её конфигурируемые модули;
- обучение, инференс и вычисление метрик через существующие точки входа;
- воспроизводимый запуск хотя бы одного базового сценария из статьи с честно задокументированными отклонениями.

## Цели разработки

- Построить воспроизводимую открытую реализацию VQA-GNN внутри текущего репозитория.
- Сохранить совместимость с существующим шаблоном запуска, конфигурации, логирования и сохранения артефактов.
- Воспроизвести базовый экспериментальный контур статьи без неявных упрощений и без подмены архитектуры.
- Подготовить кодовую базу к открытой публикации с явной фиксацией ограничений, допущений и недостающих частей.

## Карта репозитория

Ключевые части текущего репозитория:

- `train.py` — основной вход для обучения. Использует Hydra c `config_path="src/configs"` и `config_name="baseline"`.
- `inference.py` — основной вход для инференса. Использует Hydra c `config_name="inference"` и сохраняет предсказания в `data/saved/<save_path>/`.
- `requirements.txt` — текущие зависимости. База проекта построена вокруг PyTorch, Hydra, TorchMetrics, W&B и CometML.
- `.pre-commit-config.yaml` — hooks для `black`, `isort`, `flake8` и базовых проверок файлов.
- `.flake8` — текущие ограничения стиля, включая `max-line-length = 100`.
- `LICENSE` — лицензия MIT.

Структура `src/`:

- `src/configs/` — Hydra-конфигурации.
- `src/configs/baseline.yaml` — базовый training config, который собирает `model`, `writer`, `metrics`, `datasets`, `dataloader`, `transforms`, а также содержит `optimizer`, `lr_scheduler`, `loss_function`, `trainer`.
- `src/configs/inference.yaml` — базовый inference config.
- `src/configs/model/` — конфиги модели. Сейчас есть только `baseline.yaml` для `src.model.BaselineModel`.
- `src/configs/datasets/` — конфиги наборов данных. Сейчас это только примерные `example*.yaml`.
- `src/configs/dataloader/` — конфиги `DataLoader`.
- `src/configs/metrics/` — конфиги метрик.
- `src/configs/transforms/` — конфиги batch и instance transforms.
- `src/configs/writer/` — конфиги для `WandBWriter` и `CometMLWriter`.

- `src/model/` — модели. Сейчас содержит только демонстрационный `BaselineModel` в `baseline_model.py`.
- `src/datasets/` — базовый класс датасета, пример синтетического датасета, collate-функция и утилиты для dataloader.
- `src/trainer/` — обучающий контур.
- `src/trainer/base_trainer.py` — общий цикл обучения, валидации, сохранения чекпойнтов и загрузки весов.
- `src/trainer/trainer.py` — обработка batch для обучения.
- `src/trainer/inferencer.py` — инференс и сохранение предсказаний.
- `src/loss/` — функции потерь. Сейчас есть только `ExampleLoss`.
- `src/metrics/` — базовая обвязка метрик и `MetricTracker`.
- `src/transforms/` — примеры instance и batch transforms.
- `src/logger/` — логирование в консоль/файл и writer-адаптеры для W&B и CometML.
- `src/utils/` — общие утилиты.
- `src/utils/io_utils.py` — задаёт `ROOT_PATH` и содержит JSON I/O helpers.
- `src/utils/init_utils.py` — сохраняет `config.yaml`, `git_commit.txt`, `git_diff.patch`, управляет `run_id` и инициализирует logging.

Наблюдения по текущему состоянию:

- `README.md` присутствует и описывает pipeline, Kaggle, CometML, deviations.
- `CITATION.cff` отсутствует.
- Реализация VQA-GNN присутствует: `VQAGNNModel`, `VQADataset`, `VQADemoDataset`,
  `VQALoss`, `VQAAccuracy`, preprocessing scripts, configs.
- Реальные VQA-датасеты подключены через `src/datasets/vqa_dataset.py` и
  `src/configs/datasets/vqa.yaml`, но требуют заранее подготовленных артефактов.
- Demo smoke-test (`VQADemoDataset`) работает, но всегда загружает `roberta-large`.
- Числовые результаты статьи **не подтверждены** реальным end-to-end запуском.
- Каталоги `data/` и `saved/` не закоммичены, но ожидаются кодом во время выполнения.

## Существующие соглашения и точки опоры

- Точки входа уже существуют: `python train.py` и `python inference.py`.
- Конфигурация построена через Hydra и `hydra.utils.instantiate`.
- Имена модулей и файлов — в `snake_case`, классы — в `PascalCase`.
- Экспортируемые классы обычно пробрасываются через `__init__.py` пакетов.
- Код снабжён инженерными docstring-комментариями на английском.
- Batch-ключи должны совпадать между `Dataset.__getitem__`, `collate_fn`, `model.forward`, `loss.forward` и metric wrappers.
- Сохранение артефактов уже реализовано через `src/utils/init_utils.py`; нельзя дублировать этот механизм в новой ветке кода.

## Правила для Claude Code

- Использовать существующие точки входа `train.py` и `inference.py`, а не создавать альтернативные training/inference-скрипты без крайней необходимости.
- Переиспользовать существующие Hydra-группы в `src/configs/`; новые конфиги добавлять в уже существующие разделы.

- Не дублировать уже существующие abstractions из `src/trainer/`, `src/logger/`, `src/utils/` и `src/datasets/data_utils.py`.

- Не выдавать демонстрационные компоненты шаблона за части VQA-GNN.
- Не подделывать результаты, отсутствующие зависимости, доступность данных или полноту воспроизведения статьи.
- Любое отклонение от оригинальной статьи документировать явно и рядом с местом использования: в конфиге, в docstring, в README после его появления и в отчёте о запуске.
- Сохранять согласованность интерфейсов. Если меняется структура batch, одновременно проверять `src/datasets/*`, `src/datasets/collate.py`, `src/model/*`, `src/loss/*`, `src/metrics/*`, `src/trainer/*`.

## Правила воспроизводимости

- Любой недостающий компонент статьи нужно отмечать честно, а не закрывать молчаливой заменой.
- Любое допущение о данных, preprocessing, knowledge source, feature extractor, tokenizer или формате графа нужно фиксировать в документации и конфиге.
- Нужно сохранять воспроизводимость через уже существующий механизм сохранения артефактов в директории запуска.
- Обязательные артефакты воспроизводимости: `config.yaml`, `git_commit.txt`, `git_diff.patch`.
- Все запускаемые эксперименты должны иметь явный `run_name`, осмысленные значения seed и конфиги, достаточные для повторного запуска из корня репозитория.
- Если воспроизведение точных результатов статьи невозможно из-за отсутствия закрытых данных, приватных признаков или неописанных деталей, это должно быть указано напрямую.
- Код должен быть пригоден для открытой публикации: без скрытых путей, без локальных хаков, без неочевидных ручных шагов.

## Правила работы с данными

- Не пытаться автоматически скачивать закрытые, лицензируемые или требующие ручного согласия данные.
- Не повторять поведение `ExampleDataset`, который при отсутствии данных синтезирует примерные объекты на диск, для реальных VQA-ресурсов статьи.
- Для реальных датасетов использовать явные проверки наличия файлов и понятные сообщения об ошибках.
- Пути к данным должны задаваться через конфиги и индексные структуры, а не жёстко в коде.
- Если для базового запуска нужен минимальный или демонстрационный режим, он должен быть явно помечен как демонстрационный и не должен маскироваться под эксперимент статьи.
- Если статья требует внешних знаний или дополнительных артефактов, необходимо отдельно описывать ожидаемый ресурс, его расположение, требуемую версию или формат и возможность открытой публикации инструкции по подготовке.

Приоритетный порядок работ для Claude Code:

1. Зафиксировать соответствие между компонентами статьи и текущими точками расширения шаблона.
2. Подготовить реальные конфиги VQA-GNN внутри существующей Hydra-структуры, не ломая `baseline.yaml` и `inference.yaml`.
3. Реализовать интерфейсы данных VQA в `src/datasets/` и согласовать их с `collate_fn`.
4. Реализовать модель VQA-GNN в `src/model/` и экспортировать её через пакет.
5. Подключить loss и metrics, необходимые для базового ответа на вопрос, не создавая параллельный training stack.
6. При необходимости минимально адаптировать `src/trainer/` только там, где этого требует статья.
7. Подготовить воспроизводимые конфиги запуска обучения и инференса для базового сценария статьи.
8. Обновить пользовательскую документацию проекта, когда появится первый честно запускаемый контур VQA-GNN.

Что считать приоритетом на первой стадии:

- корректную структуру данных и batch-интерфейс;
- базовую модель статьи;
- совместимость с текущим train/inference pipeline;
- честную документацию ограничений воспроизводимости.

Что не является приоритетом на этой стадии:

- любые архитектурные улучшения;
- любые исследовательские ответвления за пределами исходного VQA-GNN;
- косметические рефакторинги инфраструктуры без необходимости для воспроизведения статьи.

## Критерии готовности

Воспроизводимая реализация VQA-GNN считается готовой, когда одновременно выполнены следующие условия:

- в репозитории есть реальная модель VQA-GNN, а не шаблонный `BaselineModel`, для рабочего контура статьи;
- реальные интерфейсы данных подключены через существующий `src/datasets/` и `src/configs/datasets/`;
- запуск обучения и инференса выполняется через существующие `train.py` и `inference.py`;
- конфиги VQA-GNN находятся в существующей Hydra-структуре и не требуют отдельного неформального bootstrap-кода;
- чекпойнты, конфиги и git patch сохраняются текущими средствами репозитория;
- воспроизводится как минимум один базовый экспериментальный контур статьи;
- все отклонения от статьи, ограничения данных и нерешённые расхождения явно задокументированы;
- кодовая база остаётся пригодной для открытой публикации и не зависит от скрытых локальных настроек.
